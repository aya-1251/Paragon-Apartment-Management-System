# Paragon Apartment Management System

## Agile Development Joint Report 

---

## Methodology: Scrum Framework

We selected Scrum as the most appropriate development methodology for this project due to our team size of five, which aligns with Scrum’s recommended team structure, and the complexity of the system, which consisted of six distinct components.

---

## Scrum Roles

Within the team, each member had a defined Scrum role alongside their development responsibilities:

* Aya: The **Product Owner** was responsible for managing the product backlog, prioritising user stories, and ensuring alignment with stakeholder requirements.
* Isam: The **Scrum Master** facilitated sprint planning, standups, reviews, and retrospectives, while also removing blockers and ensuring Scrum practices were followed.
* All five: The **Developers** were responsible for designing, implementing, and testing the system modules across all three sprints.

---

## Scrum Events

We consistently followed core Scrum events across all three sprints:

* **Sprint Planning:** At the start of each sprint, we selected user stories, defined sprint goals, and distributed tasks based on strengths and responsibilities.
* **Daily Standups:** We conducted brief asynchronous updates via group chat, where each of us shared progress, upcoming tasks, and any blockers.
* **Sprint Review:** At the end of each sprint, we demonstrated completed features and evaluated whether the sprint goal had been achieved.
* **Sprint Retrospective:** We reflected on what worked well and what could be improved, then applied those improvements in the next sprint.

---

## Scrum Artefacts

Throughout the project, we maintained key Scrum artefacts:

* **Product Backlog:** A prioritised list of user stories derived from the Paragon requirements.
* **Sprint Backlogs:** A subset of selected tasks for each sprint, with effort estimates.
* **Increments:** At the end of each sprint, we delivered a working system increment that met our Definition of Done (implemented, tested, and reviewed).

---

## Sprint Overview

* **Sprint 1:** Focused on requirements analysis, system design, and building the foundation, including user authentication and role-based access control.
* **Sprint 2:** Focused on developing core modules such as tenant management, apartment management, and billing.
* **Sprint 3:** Focused on completing remaining modules, full system integration, testing, and bug fixing.

Each sprint resulted in a functional increment that built toward the final system.

---

## Steps Followed in the Agile Development Episode

### Step 1: Requirements Analysis & Product Backlog Creation

We began by analysing the Paragon problem statement and breaking it down into user stories covering all six system components. These were prioritised based on dependencies and business value.

Example user stories included:

* *As a Manager, I should be able to view occupancy levels across all cities to monitor performance.*
* *As a Finance Manager, I should be able to generate financial summaries to manage cash flow.*

This step was essential in structuring the development process and ensuring clarity before implementation.

---

### Step 2: Sprint 1 — Design & Core Foundation

In Sprint 1, we focused on designing the system architecture and building the Account/User Management module. This included database design, UI wireframes, and implementing a login system with role-based access control.

By the end of the sprint, we had a working authentication system supporting five user roles, which formed the foundation for the rest of the system.

---

### Step 3: Sprint 2 — Core Modules Development

During Sprint 2, we developed the core system modules:

* Tenant Management
* Apartment Management
* Payment & Billing

We implemented key features such as tenant registration, lease management, apartment assignment, and billing logic, including late payment tracking and notifications.

At the end of the sprint, we conducted integration testing to ensure these modules worked together correctly.

---

### Step 4: Sprint 3 — Remaining Modules, Integration & Testing

In the final sprint, we completed the remaining components:

* Maintenance module
* Report Generation

We implemented maintenance workflows (issue reporting, assignment, tracking) and generated reports such as occupancy, financial summaries, and maintenance costs.

We then completed full system integration and conducted end-to-end testing. Bugs identified during testing were fixed and re-tested within the sprint, resulting in a fully functional system that met all acceptance criteria.

---

### Step 5: Sprint Reviews & Retrospectives

After each sprint, we held reviews and retrospectives, which helped improve our process:

* After Sprint 1, we improved task granularity to make work easier to track.
* After Sprint 2, we increased integration testing during development rather than leaving it until the end.
* After Sprint 3, we ensured documentation was completed alongside development.

These improvements made our workflow more efficient and contributed to a smoother final delivery.
